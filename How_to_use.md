# Latebris 사용 방법

## 1. 문서 목적

Latebris는 안드로이드 위치 환경을 시뮬레이션하기 위한 내부 QA 도구입니다.

- 앱 내부 프리셋을 기반으로 GPS 테스트 데이터를 사용할 수 있습니다.
- Wi-Fi 및 Cell 데이터도 프리셋 기반으로 다루지만, 이 부분은 Xposed/LSPosed 후킹 경로가 필요합니다.
- 이 프로젝트는 사내 테스트 기기와 개발/QA 환경을 전제로 하며, 일반 배포용 앱이 아닙니다.

---

## 2. 시작 전에 확인할 것

### 기본 기기 조건

- Android 12 이상
- APK가 기기에 설치되어 있어야 함
- 위치 권한이 허용되어 있어야 함
- Android 13 이상에서는 알림 권한도 허용되어 있어야 함

### 전체 기능 사용 시 추가 조건

Wi-Fi / Cell 모킹까지 포함해 전체 기능을 사용하려면 아래 환경이 추가로 필요합니다.

- 루팅된 기기
- Magisk
- LSPosed 또는 호환되는 Xposed 런타임

LSPosed가 없더라도 앱 UI, 프리셋 관리, 시뮬레이션 제어 흐름은 확인할 수 있지만, Wi-Fi / Cell 후킹 동작은 활성화되지 않습니다.

---

## 3. 사용 모드

### 모드 A. 일반 Android 폰 기준 사용

이 모드는 다음을 확인할 때 적합합니다.

- 프리셋 생성 및 편집
- 대시보드에서 시뮬레이션 시작/중지
- 서비스 상태 저장 및 UI 반영 흐름

### 모드 B. Root + LSPosed 사용

프리셋 기반 Wi-Fi / Cell 데이터 치환까지 확인하려면 이 모드가 필요합니다.

현재 구조는 시스템 전역 값을 바꾸는 방식이 아니라, **LSPosed에서 scope에 추가한 앱 프로세스 안에서만** Wi-Fi / Cell API 결과를 프리셋 값으로 바꾸는 방식입니다.

즉, 외부 테스트 대상 앱을 검증하려면 LSPosed에서 그 앱을 이 모듈의 scope에 직접 추가해야 합니다.

---

## 4. 초기 설정

### 4-1. 앱 설치

테스트 기기에 APK를 설치합니다.

현재 프로젝트의 debug APK 예시 경로:

- `app/build/outputs/apk/debug/app-debug.apk`

### 4-2. 앱 첫 실행

앱을 처음 실행하면 내부 프리셋 저장소가 초기화됩니다.

- `assets/presets/default-presets.json`의 기본 프리셋이 내부 저장소로 복사됩니다.
- 이후 대시보드는 내부 저장소의 프리셋 목록을 기준으로 동작합니다.

### 4-3. 런타임 권한 허용

앱을 실행한 뒤 상단 상태 카드와 권한 상태를 확인합니다.

필요 시 다음 권한을 허용합니다.

- 위치 권한
- 알림 권한(Android 13 이상)

권한이 없으면:

- 대시보드에 **Action required** 상태가 표시될 수 있고
- **Start** 버튼이 비활성화되며
- **Request permissions** 버튼으로 권한 요청을 진행할 수 있습니다.

---

## 5. 메인 화면 구성

메인 화면은 크게 세 부분으로 구성됩니다.

### 상태 카드

다음 정보를 표시합니다.

- 현재 시뮬레이션 상태: `Running` 또는 `Stopped`
- 권한 준비 상태
- 현재 선택된 프리셋 또는 활성 프리셋
- 실행 중인 경우 활성 위도/경도

### 프리셋 목록

- 항목을 탭하면 해당 프리셋이 선택됩니다.
- 선택된 프리셋은 하이라이트됩니다.
- **Edit** 버튼으로 기존 프리셋을 수정할 수 있습니다.
- **New preset** 버튼으로 새 프리셋을 만들 수 있습니다.

### 서비스 제어 영역

- **Start**: 선택한 프리셋으로 시뮬레이션을 시작합니다.
- **Stop**: 현재 시뮬레이션을 중지합니다.

---

## 6. 기본 사용 순서

### 1단계. 앱 실행

상태 카드에서 아래 항목을 먼저 확인합니다.

- permission readiness = `Ready`

### 2단계. 프리셋 선택

목록에서 원하는 프리셋 하나를 선택합니다.

Start가 활성화되려면 권한뿐 아니라 **유효한 프리셋이 하나 선택되어 있어야** 합니다.

### 3단계. 시뮬레이션 시작

**Start** 버튼을 누릅니다.

내부적으로는 다음 순서로 동작합니다.

- `SimulatorControlService` 시작
- 현재 활성 프리셋 ID 및 실행 상태 저장
- `GpsMockService` 시작
- foreground service 기반 시뮬레이션 흐름 활성화

### 4단계. 실행 상태 확인

정상적으로 시작되면 대시보드에 다음이 표시됩니다.

- `Running`
- 활성 프리셋 이름
- 활성 위도 / 경도

### 5단계. 시뮬레이션 중지

**Stop** 버튼을 누릅니다.

중지 시 내부적으로는 다음이 수행됩니다.

- `GpsMockService` 중지
- 시뮬레이션 상태 초기화
- 공유 상태가 off로 바뀌면서 Wi-Fi / Cell 후킹도 fallback 경로로 돌아감

---

## 7. 프리셋 편집 방법

프리셋 편집 화면은 다음 두 경로로 들어갈 수 있습니다.

- **New preset**: 새 프리셋 생성
- **Edit**: 기존 프리셋 수정

### 수정 가능한 항목

#### 기본 정보

- Name
- Latitude
- Longitude
- Altitude

#### Wi-Fi 정보

각 Wi-Fi 행에서 아래 값을 편집할 수 있습니다.

- `SSID`
- `BSSID`
- `Level`
- `Frequency`

#### Cell 정보

각 Cell 행에서 아래 값을 편집할 수 있습니다.

- `MCC`
- `MNC`
- `CI`
- `PCI`
- `TAC`
- `EARFCN`

### 입력 검증 규칙

저장 가능한 상태가 아니면 Save/Create가 비활성화됩니다.

- Name은 필수입니다.
- Latitude는 `-90 ~ 90` 범위여야 합니다.
- Longitude는 `-180 ~ 180` 범위여야 합니다.
- Altitude는 숫자여야 합니다.
- 추가한 Wi-Fi / Cell 행은 비워 두거나, 모두 완전히 입력해야 합니다.

### 저장 / 삭제

- **Create** 또는 **Save**를 누르면 프리셋이 내부 저장소에 반영됩니다.
- **Delete**를 누르면 해당 프리셋이 삭제됩니다.

---

## 8. Cell Auto Fill 사용법

프리셋 편집 화면의 **Cell towers** 섹션에는 **Auto Fill** 버튼이 있습니다.

### 동작 방식

- 현재 편집 중인 위도 / 경도를 사용합니다.
- OpenCellID에 요청을 보냅니다.
- 응답받은 주변 셀 정보를 편집 화면의 `cellTowers`에 채웁니다.

### 중요한 점

Auto Fill은 **현재 편집 중인 draft**만 갱신합니다.

- Auto Fill 결과는 **Save를 눌러야만** 실제 저장됩니다.

### 필요 조건

실제 외부 셀 데이터를 받으려면 빌드에 아래 값이 포함되어 있어야 합니다.

- `OPEN_CELL_ID_API_KEY`

또한 현재 편집 중인 프리셋 상태가 유효해야 합니다. 이름, 좌표, 고도, Wi-Fi / Cell 입력 상태가 검증을 통과하지 못하면 Auto Fill 요청도 진행되지 않습니다.

즉, API 키가 없거나 편집 상태가 유효하지 않으면 Auto Fill은 외부 데이터를 채우지 못하고 현재 편집 상태를 그대로 유지합니다.

---

## 9. Root / LSPosed 설정

Wi-Fi / Cell 후킹까지 포함해 전체 테스트를 하려면 아래 순서를 따릅니다.

1. 기기를 루팅합니다.
2. Magisk를 설치합니다.
3. LSPosed를 설치합니다.
4. LSPosed에서 이 모듈을 활성화합니다.
5. LSPosed에서 **테스트 대상 앱을 이 모듈의 scope에 추가합니다.**
6. LSPosed 환경에서 필요하면 재부팅합니다.
7. Latebris 앱을 열고 프리셋을 선택한 뒤 시뮬레이션을 시작합니다.

### 매우 중요한 점: 테스트 대상 앱 scope 추가

이 단계가 빠지면 외부 앱에서 Wi-Fi / Cell 값이 바뀌지 않습니다.

권장 순서:

1. LSPosed Manager를 엽니다.
2. **Modules** 탭에서 `Latebris` 모듈을 엽니다.
3. **Enable module**이 켜져 있는지 확인합니다.
4. 오른쪽 상단 검색 또는 앱 목록에서 **실제로 테스트할 앱**을 찾습니다.
5. 그 앱을 체크해서 scope에 추가합니다.
6. 필요 시 앱을 강제 종료 후 다시 실행하거나 기기를 재부팅합니다.

예시:

- 사내 테스트 대상 앱이 `com.company.app`이면, LSPosed scope에 `com.company.app`을 추가해야 합니다.
- `Latebris` 앱만 켜 두고 대상 앱을 scope에 넣지 않으면, 외부 앱 호출 결과는 바뀌지 않습니다.

### 현재 동작 방식

- GPS는 mock location 기반이라 시스템 위치 테스트 흐름에 사용됩니다.
- Wi-Fi / Cell은 LSPosed 후킹 기반입니다.
- 후킹은 scope에 추가된 앱의 프로세스에서만 적용됩니다.
- 시뮬레이션이 꺼져 있거나 해당 기능 토글이 꺼져 있으면 원래 API 결과를 그대로 사용합니다.

### 현재 프로젝트에서 후킹하는 대상

현재 모듈은 아래 메서드를 후킹합니다.

- `WifiManager.getScanResults()`
- `TelephonyManager.getAllCellInfo()`
- `TelephonyManager.getCellLocation()`

후킹 모듈은 아래 공유 상태를 읽습니다.

- 현재 활성 프리셋 ID
- 시뮬레이션 on/off 상태

아래 조건이 만족될 때만 프리셋 기반 Wi-Fi / Cell 데이터를 반환합니다.

- 시뮬레이션이 running 상태일 것
- 활성 프리셋이 있을 것
- 해당 프리셋에 `wifiNetworks` 또는 `cellTowers` 값이 있을 것

그 외에는 원래 앱/시스템 결과로 fallback 됩니다.

## 10. 현재 동작상 알아둘 점

- 시뮬레이션이 실행 중이면 프리셋 선택과 편집이 잠깁니다.
- 앱 시작 시 실제 서비스가 살아 있지 않으면 stale running state를 자동 정리합니다.
- Android 13 이상에서는 알림 권한 상태도 UI에 표시됩니다.
- 현재 Wi-Fi / Cell 후킹 범위는 `com.example.gpstick` 패키지로 제한됩니다.
- LSPosed upstream은 공식적으로 Android 14까지 지원한다고 문서화되어 있으므로, 그 이후 버전은 미확인 영역으로 보는 것이 안전합니다.

---

## 11. 권장 첫 테스트 시나리오

### 일반 Android 기기 기준 간단 테스트

1. 앱을 설치합니다.
2. 앱을 열고 필요한 권한을 허용합니다.
3. 기본 프리셋 하나를 선택합니다.
4. **Start**를 누릅니다.
5. 상태가 `Running`으로 바뀌고 활성 좌표가 보이는지 확인합니다.
6. **Stop**을 누릅니다.
7. 상태가 `Stopped`로 돌아오는지 확인합니다.

### Root + LSPosed 포함 전체 테스트

1. 위 기본 테스트를 먼저 완료합니다.
2. LSPosed에서 모듈을 활성화합니다.
3. 필요 시 재부팅합니다.
4. 다시 시뮬레이션을 시작합니다.
5. `com.example.gpstick` 내부에서 Wi-Fi / Cell API 결과가 프리셋 기반으로 바뀌는지 확인합니다.

## 12. 문제 해결

### Start 버튼이 비활성화되어 있을 때

다음을 확인합니다.

- 프리셋이 선택되어 있는지
- 위치 권한이 허용되어 있는지
- Android 13+에서 알림 권한이 허용되어 있는지
- 이미 시뮬레이션이 실행 중은 아닌지

### Auto Fill이 동작하지 않을 때

다음을 확인합니다.

- 위도 / 경도가 유효한 값인지
- 빌드에 `OPEN_CELL_ID_API_KEY`가 포함되어 있는지
- 기기에 네트워크 연결이 있는지

### Wi-Fi / Cell 값이 바뀌지 않을 때

다음을 확인합니다.

- 기기가 루팅되어 있는지
- LSPosed가 설치 및 활성화되어 있는지
- 모듈이 활성화되어 있는지
- 현재 시뮬레이션이 running 상태인지
- 활성 프리셋에 Wi-Fi / Cell 항목이 실제로 들어 있는지
- 테스트 중인 대상이 `com.example.gpstick`인지

### 앱을 다시 열었는데 상태가 이상할 때

앱을 한 번 다시 열어 보세요. 앱 시작 시 stale state 정리 로직이 동작하므로, 실제 서비스가 없는 잘못된 running 상태는 자동으로 초기화되어야 합니다.
